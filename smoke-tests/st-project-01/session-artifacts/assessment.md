Prioritized findings:

1. The top risk is silent corruption of technical prose. The current period-spacing and sentence-capitalization heuristics in [src/smoother.ts](/Users/leemoore/code/agent-cli-tools/cxs-cloner/smoke-tests/st-project-01/src/smoother.ts) turn `example.com` into `example. Com`, `config.json` into `config. Json`, and even change inline-code `` `i` `` to `` `I` ``. That breaks the project’s “conservative deterministic cleanup” positioning.

2. The 1 MB claim is not currently credible for punctuation-dense input. The logic at [src/smoother.ts](/Users/leemoore/code/agent-cli-tools/cxs-cloner/smoke-tests/st-project-01/src/smoother.ts) repeatedly slices from the start of the string while scanning periods, which behaves quadratically. In a local spot check, 16 KB took about 213 ms, 32 KB about 831 ms, 64 KB about 3.3 s, 128 KB about 13.2 s, and a 1 MB synthetic run did not finish within 45 seconds.

3. CLI UX is mostly clean, but it silently accepts conflicting input sources. In [src/cli.ts](/Users/leemoore/code/agent-cli-tools/cxs-cloner/smoke-tests/st-project-01/src/cli.ts), `--input` simply overrides a positional path with no warning, which is surprising for automation. Stats are also human-readable only, so they’re harder to consume in scripts.

4. Maintainability is fine for a tiny v1, but the architecture will resist safe growth. The rule pipeline, stats accounting, punctuation heuristics, and allowlists are all tightly packed into one module, and similar boundary logic is duplicated instead of shared. The tests in [tests/smoother.test.ts](/Users/leemoore/code/agent-cli-tools/cxs-cloner/smoke-tests/st-project-01/tests/smoother.test.ts) and [tests/cli.test.ts](/Users/leemoore/code/agent-cli-tools/cxs-cloner/smoke-tests/st-project-01/tests/cli.test.ts) cover happy-path prose well, but miss domains, emails, filenames, inline code, idempotence on mixed technical text, conflicting CLI input, and scale guardrails.

The follow-up plan I recommended is: fix safety boundaries first with protected spans for URLs/emails/code-like tokens, unify period-boundary logic into one classifier, make that path linear-ish for 1 MB inputs, then tighten the CLI contract and add boundary-focused tests.
